# 標題公式庫

> Step 1 產候選時用。**強制跨公式**（不准 6 個都是 how-to）。
> 每個公式給：結構 + 何時用 + 範例（含一個技術部落格情境）。

## 1. 數字式（最強，有數字優先用）

**結構**：`數字 + 名詞 + 收穫/結果`
**何時用**：文章有可量化的東西（篇數 / 倍數 / 秒數 / 行數 / 天數 / 坑數）。
**為什麼有效**：數字=具體性=可信。奇數 / 非整數("44 篇""3.5 倍")比整數("40 篇""3 倍")更可信(BuzzFeed +20% 點擊)。

範例：
- `44 篇舊文一次改 SEO：我用 AI 批次重構的踩坑覆盤`
- `從 8 秒到 0.3 秒：這個查詢我改了 3 個地方`

## 2. 痛點 / 張力式

**結構**：`點出讀者正在受的苦 + 暗示有解`
**何時用**：文章解決一個明確、煩人、反覆出現的問題。
**為什麼有效**：痛點直接命中搜尋意圖；張力製造非讀不可的理由。

範例：
- `git push 完還在手動盯 Actions 轉圈？我讓 Claude 自己驗部署`
- `RecyclerView 滑起來卡頓，問題不在你以為的地方`

## 3. 反直覺 / 結果式

**結構**：`一個違反常識的結論 / 出乎意料的結果`
**何時用**：文章的核心發現跟大家預設不一樣。
**為什麼有效**：認知衝突製造好奇缺口(curiosity gap)，逼人點進來解惑。

範例：
- `升級 Spring Boot 4 之後，我反而把一半的新功能關掉了`
- `寫了測試覆蓋率 100%，線上還是炸了——問題出在哪`

## 4. How-to / 教學式（最常見，但要加料）

**結構**：`怎麼做 X（+ 限定條件讓它不像通用模板）`
**何時用**：純操作教學。**但裸 how-to 是爛標題的重災區**，必須加一個差異化元素：數字 / 時間 / 條件 / 反差。
**為什麼要加料**：「X 怎麼用？完整教學」跟其他 1000 篇撞臉，沒有點擊理由。

範例：
- ❌ `Claude Code Hook 怎麼用？完整教學`（裸 how-to，爛）
- ✅ `用 Claude Code Hook 自動驗部署：90 行 shell + 一次性排程不輪詢`（加了行數 + 機制差異）

## 5. 問句式

**結構**：`一個讀者心裡正在問的問題`
**何時用**：搜尋意圖本身就是疑問句（「值不值得」「該不該」「哪個好」）。
**注意**：Backlinko 數據顯示問句 vs 非問句 CTR 差異可忽略——**別為了問而問**，只在問句剛好等於搜尋意圖時用。

範例：
- `Spring Boot 4 跟 3，升級到底值不值得？`
- `MSSQL 跑 30 秒的查詢，加索引就夠了嗎？`

## 6. APP 公式（Agree-Promise-Preview，Brian Dean）

**結構**：`先講一個讀者會認同的前提 → 承諾解法 → 預告內文有什麼`
**何時用**：稍長的標題或副標 / meta description，需要鋪陳說服。
**範例**：`大家都說 RAG 很簡單（Agree）。但要做到能上線（Promise），這 3 個坑沒人講（Preview）`

## 7. AIDA 公式（Attention-Interest-Desire-Action）

**結構**：`抓注意 → 勾興趣 → 激起渴望 → 暗示行動`
**何時用**：偏行銷 / 推廣性質的文，或寫 meta description 時。
**範例**：標題抓注意「我把發文流程全自動化了」，meta 接興趣+渴望+行動「從查證到發布 6 步，附完整 skill code，照著做就能複製」。

---

## 組合技

好標題常是**多公式疊加**：
- 數字 + 痛點：`還在手動跑 5 個步驟發文？我用 1 個 skill 收掉了`
- 反直覺 + 數字：`關掉 3 個「最佳實踐」之後，build 快了一倍`
- How-to + 結果：`用 commit SHA 對齊驗 CI：換任何 repo 都不用改`

## 選公式的決策樹

```
有具體數字嗎？
├─ 有 → 優先「數字式」或「數字+X 組合」
└─ 無 → 文章是解決痛點嗎？
        ├─ 是 → 「痛點/張力式」
        └─ 否 → 有反直覺結論嗎？
                ├─ 有 → 「反直覺/結果式」
                └─ 無 → 「How-to 加料」或「問句（僅當=搜尋意圖）」
```
